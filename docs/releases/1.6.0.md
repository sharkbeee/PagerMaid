# PagerMaid 1.6.0 Release Notes

PagerMaid 1.6.0 completes the planned web admin hardening and refactor track.
It also updates project ownership references, install sources, telemetry defaults,
and locally served web branding assets.

## Highlights

### Web Admin Architecture

- Added a typed web settings boundary (`9858a80`).
- Introduced a FastAPI app factory and centralized router assembly
  (`4db6e27`, `ae18f04`, `eb3c47c`).
- Added centralized web exception handler registration (`0a365a9`).
- Added characterization and settings tests to preserve expected behavior
  (`e77ff05`, `ee4c8c7`).

### Authentication and Sessions

- Replaced frontend token storage with HTTP-only cookie sessions
  (`17060d1`, `891267e`).
- Added explicit session-check and logout endpoints (`8c58799`).
- Removed browser `localStorage` authentication (`b6bdb7f`).
- Added configurable session and cookie settings (`3ff23fe`).

### Dangerous Operations

- Disabled web `shell` and `eval` by default and removed them from the normal
  admin UI (`398b1e5`).
- Added tests covering authentication gates and disabled-by-default behavior
  (`79b564c`).

### Reliability and Deployment

- Added web deployment guidance and migration notes (`098f2c9`, `06cf7cf`).
- Improved status handling when system metrics are unavailable (`bea3661`).
- Migrated status serialization to the Pydantic v2 API (`a477f54`).
- Served web branding assets locally instead of relying on remote assets
  (`8db7dc4`).

### Project Maintenance

- Documented the maintained fork and updated repository links and install
  sources (`bc0a3ac`, `45c1fd1`, `4c55e64`).
- Changed telemetry and error reporting defaults to require explicit operator
  configuration (`d849fa0`).

## Upgrade and Migration Checklist

1. Review [web migration notes](../web-migration-notes.md) before enabling the
   web admin.
2. Log in again after upgrading. Sessions created by the previous web
   authentication flow should be treated as invalid.
3. Rotate `WEB_SECRET_KEY` if it was previously exposed through request
   headers, browser storage, logs, or documentation.
4. For network exposure, run the admin behind HTTPS and a reverse proxy,
   configure explicit CORS origins, and review
   [web deployment guidance](../web-deployment.md).
5. Do not rely on web `shell` or `eval`; both are disabled by default and are
   treated as development-only capabilities.
6. Review telemetry configuration. Analytics and error reporting are no longer
   enabled by default.
7. Review clone, installer, Docker, and self-update sources if upgrading an
   existing deployment from the previous upstream repository.

## Compatibility Notes

The following important routes remain available:

- `/admin`
- `/login`
- `/pagermaid/api/login`
- `/pagermaid/api/status`
- command alias endpoints
- plugin list and mutation endpoints
- ignore group list and mutation endpoints

Intentional behavior changes:

- Protected APIs now use cookie sessions.
- The login response no longer exposes a reusable frontend token.
- Raw `WEB_SECRET_KEY` and the old `token` request header no longer
  authenticate protected APIs.
- Unauthenticated protected API requests return `401`.
- Web `shell` and `eval` are disabled by default.

## Deferred Follow-Up

- Evaluate removal of AMIS from the admin UI.
- Continue normal authentication hardening and security review.
- Consider a broader project-wide settings refactor after the web boundary has
  remained stable.
